Notes:

System Error: there's a problem with the system or external devices with which the progrma is interacting.

Programmer Error: the program contains incorrect sintax or faulty logic. It could even be as simple as a typo.

User Error: the user has entered data incorrectly, which the program is unable to handle.


We can't prevent some of the system errors, while the programmer errors are solely our responsibility. The user errors mught be our mistake too, if the desigg isn't clear enough to the user.

Exeptions: an exeption is an error that produces a return value that can then be used by the program to deal with the error. I.E: calling a method that doesn't exist.

Stack trace is the path the program followed until it encountered the exeption.

Warnings: Are produced by an error that doesn't make the program crash right away. The problem can arise later when the program needs the 'thing' that failed (for example, a variable).

As Javascript is very flexible, we need to make it fail loudly in development, so any error can be spotted and fixed quicly. In production we need to make it fail gracefuly, so it still throws the error but doesn't interrupt the user.


Strict Mode: to use it you have to write 'use strict'; in the first line of the JS file. Strict mode can also be used inside a specific function. The recommended way is to use a self invoking function. This is useful if you're working in somebody else's code and you don't know if they've been coding in strict mode.


USE a linting tool, such as JSLint

Use feature detection to check if the browser have certain features. This can be done with a simple if statement and the name of the API or funtion. If we don't check beforehand, we will get an error. Checking is failing graciously :)

Debugging in the Browser:

Breakpoint allow us to follow the program step by step and check the values of our variables.

Using alerts and logging to the console are useful tools to check what's happening inside our program.

Another way is using the keyword debugger in the JS code, which will pause the execution of the program so we can check it.

Error Objects:

We can create our own error objects (messages) using the error constructor. const error = new Error();. So, in the parenthesis we can write our error message and have it personalized.

Using the throw keyword, followed by our error, we can throw our own errors.

ExceptionHandling:

When an exception occurs, the program terminates with an error message. To the user it will look like if the program crashes. What we can do is to "catch" the error, and keep the program running. This is useful in places of the code where we know there might be errors (like when we are connecting to a third party API, or doing file I/O).

try, catch, finally. We should first "try" to do something. If that fails we "catch" the error. "Finally", we continue our program.

Tests:

We can write and perform tests for our program. A test can be as simple as a function that gives a certain input and compares the output the program gives against the output the program should be giving.

Test Driven Development:

Is the process of writing tests before writing the code. Once we have the tests written we write the code that will pass these tests successfully. Then the code is continuosly improved and tested again.

TDD is considered best practice, but most of the time programmers are more pragmatic. Tests are written, of course, but the process is more dinamic.

Test-driven Development Workflow:

  1. Write tests (that initially fail).
  2. Write code to pass the tests.
  3. Refactor the code.
  4. Test refactored code.
  5. Write more tests fowr new features.

Jest:

Jest is a test framework. Instead of writing all our tests ourselves, we can use Jest. Jest provides helper methods for common test assertions.

test('square root of 4 is 2', () => {
expect(squareRoot(4)).toBe(2);
});
